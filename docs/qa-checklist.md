# QA 체크리스트

이 문서는 `docs/checklist.md`의 이슈 단위로 브라우저, 네트워크 탭, Supabase, 터미널에서 직접 확인할 수 있는 QA 절차를 정리한다.

공통 개발자 확인:
- [ ] `npx tsc --noEmit`이 에러 없이 통과한다.
- [ ] `npm run test:run`이 에러 없이 통과한다.
- [ ] 변경 범위가 화면에 영향을 주면 `npm run build`가 통과한다.
- [ ] `npm run dev` 후 `http://localhost:3000`에서 콘솔 빨간 에러가 없는지 확인한다.
- [ ] 빌드 후 dev 서버를 계속 쓸 경우 `.next` 캐시를 지우고 dev 서버를 재시작한 뒤 확인한다.

## Feature 1: 프로젝트 초기 세팅

### Next.js 14 프로젝트 초기화 및 폴더 구조 설정
- [ ] `src/app/page.tsx`가 정상 렌더링되고 `http://localhost:3000`이 `200`으로 열린다.
- [ ] `@/` alias import가 최소 1개 이상 정상 동작하는지 확인한다.
- [ ] 존재하지 않는 경로 접속 시 Next 기본 404가 깨지지 않는지 확인한다.
- [ ] `src/app`, `src/components`, `src/hooks`, `src/lib`, `src/stores`, `src/types` 구조가 유지되는지 확인한다.

### Tailwind CSS + 글로벌 스타일 설정
- [ ] 홈 화면에서 `max-w-sm`, 카드 radius, gray 계열 색상이 Tailwind 스타일대로 적용되는지 확인한다.
- [ ] SVG 아이콘이 화면 전체로 커지지 않고 `w-4`, `w-5` 크기로 표시되는지 확인한다.
- [ ] 모바일 폭 375px에서 좌우 여백과 카드 너비가 깨지지 않는지 확인한다.
- [ ] CSS 링크가 Network 탭에서 `200`으로 로드되는지 확인한다.

### Supabase 프로젝트 연결 및 환경변수 설정
- [ ] `.env.local`에 `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`가 설정되어 있는지 확인한다.
- [ ] Supabase URL이 잘못됐을 때 앱이 빌드 단계에서 죽지 않는지 확인한다.
- [ ] 로그인 모달 또는 저장 기능에서 Supabase 미설정 fallback이 의도대로 동작하는지 확인한다.
- [ ] 민감한 값이 git diff에 노출되지 않는지 확인한다.

### Supabase DB 스키마 설계 및 마이그레이션
- [ ] Supabase SQL Editor에서 마이그레이션 SQL을 실행했을 때 에러 없이 완료되는지 확인한다.
- [ ] Table Editor에서 `regions`, `places`, `saved_places`, `courses` 테이블이 보이는지 확인한다.
- [ ] RLS가 필요한 테이블에서 본인 데이터만 조회/수정 가능한지 다른 계정으로 확인한다.
- [ ] FK가 있는 테이블에서 참조 대상 삭제 시 예상한 cascade 또는 제한 동작을 확인한다.

### 공통 TypeScript 타입 정의
- [ ] `Region`, `Place`, `Course`, `FilterTag` 타입을 사용하는 화면이 타입 에러 없이 빌드되는지 확인한다.
- [ ] API 응답을 타입으로 변환하는 코드에서 필수 필드 누락 시 fallback이 있는지 확인한다.
- [ ] mock 데이터가 타입과 일치하는지 `npx tsc --noEmit`으로 확인한다.
- [ ] 저장/공유 기능에서 `Place` 객체 직렬화가 깨지지 않는지 확인한다.

### ESLint / Prettier 설정 및 커밋 훅
- [ ] `npm run build`의 lint 단계가 통과하는지 확인한다.
- [ ] 새 파일의 import 순서와 unused import가 lint에서 걸리지 않는지 확인한다.
- [ ] 기존 `<img>` 경고처럼 의도적으로 남긴 경고와 신규 에러를 구분해 기록한다.
- [ ] 커밋 전에 의도하지 않은 파일이 staging되지 않았는지 `git status --short`로 확인한다.

## Feature 2: 데이터 파이프라인

- [ ] 외부 API timeout 또는 5xx 응답 시 에러 로그와 응답 상태가 확인 가능한지 확인한다.

### 지역 핫도 스코어 산출 로직 구현

- [ ] 검색량 상승, 혼잡도 증가 케이스에서 score가 상승하는지 확인한다.
- [ ] 입력값이 0이거나 누락된 경우 `NaN`이 나오지 않는지 확인한다.
- [ ] score 정렬 결과가 홈 랭킹 표시 순서와 일치하는지 확인한다.

### Vercel Cron Job 설정
- [ ] `vercel.json`의 cron path가 실제 route path와 일치하는지 확인한다.
- [ ] cron route가 `CRON_SECRET` 검증을 수행하는지 확인한다.
- [ ] 로컬에서 bearer token이 맞을 때와 틀릴 때 응답이 구분되는지 확인한다.
- [ ] Vercel 환경변수에 필요한 외부 API 키가 등록되어 있는지 확인한다.

### 카카오 로컬 API 연동 및 장소 정보 수집

- [ ] `KAKAO_REST_API_KEY`가 없을 때 mock 장소 fallback이 보이는지 확인한다.
- [ ] 카카오 응답 장소가 `Place` 타입으로 변환되어 카드 이름/주소/카테고리에 표시되는지 확인한다.
- [ ] 잘못된 `regionId` 요청 시 `400` 또는 빈 상태 UI가 표시되는지 확인한다.

### Supabase 캐싱 전략 및 API fallback 처리
- [ ] 외부 API 실패 시 마지막으로 저장된 Supabase 데이터가 반환되는지 확인한다.
- [ ] Supabase 데이터도 없을 때 mock 또는 빈 상태 UI가 표시되는지 확인한다.
- [ ] fallback 응답이 너무 오래 stale되지 않도록 갱신 기준을 확인한다.
- [ ] Network 탭에서 실패 요청이 무한 재시도되지 않는지 확인한다.
